Section 01

문제 정의

고령자는 대중교통을 이용할 때 두 종류의 장벽을 동시에 만난다. 하나는 앱 자체의 복잡함이다. 버스 앱, 지하철 앱, 길찾기 앱마다 메뉴 구조가 다르고, 원하는 기능까지 가려면 여러 화면을 거쳐야 한다. 다른 하나는 환승 구간의 막막함이다. 버스에서 내려 다음 교통수단으로 이어가야 하는 순간, 어디로 가야 하는지, 무엇을 타야 하는지 판단할 정보가 부족하다.

핵심 문제 정의 — 기존 음성 호출형 서비스(예: "차 불러줘")는 단발성 명령 처리에 머물러, 고령자가 실제로 겪는 "목적지를 정하고 → 경로를 비교하고 → 환승까지 이어가는" 연속적인 의사결정 과정을 지원하지 못한다.

본 프로젝트는 대화로 목적지를 확정하고, 지도로 경로를 함께 확인하고, 마지막 구간까지 자연스럽게 안내하는 멀티턴 이동 에이전트를 제안한다. 음성은 입력 방식 중 하나일 뿐이며, 핵심은 다단계 의도 파악과 상태 관리에 있다.

Section 02

주제 변경 배경

본 팀은 초기 투표에서 PM 고위험 지역 예측 및 알림(4명)과 고령자 어플 사용성 개선(3명)을 검토했다. 이후 공단 자문 결과, 데이터 제약이 명확해지며 주제 우선순위를 재조정했다.

공단 자문 의견 (조용빈 선임연구원)

PM 고위험 지역 예측 및 알림
"주제의 핵심이 되는 PM 이동량·사용량·사고 위치 데이터는 공단에서 제공해 드리기 어려운 데이터입니다... 다른 주제를 검토해보면 좋지 않을까 싶습니다."
고령자 앱 사용성 개선 서비스
"방향 자체는 좋습니다만, 음성 기반 DRT 호출 기능(ASR-LLM-TTS)은 현재 유사한 방향으로 추진되고 있는 사항이 있어, 이 부분만으로는 차별점이 약할 수 있습니다."

판단 — PM 주제는 핵심 데이터 자체가 제공 불가하여 진행이 어렵다고 판단했다. 고령자 주제는 "차별점이 약할 수 있다"는 지적이었으나, 이는 음성 호출이라는 입력 방식에 대한 평가였다. 처리 방식을 단발성 명령 호출에서 다단계 대화형 에이전트로 재설계하면 지적된 중복성을 해소할 수 있다고 보고 주제를 구체화했다.

기존 음성 DRT 호출형

  • "차 불러줘" → 즉시 API 호출 1회
  • 이전 대화 맥락을 기억하지 못함 (stateless)
  • 경로·교통수단 비교 기능 없음
  • 환승 구간 안내 없음
vs

본 프로젝트 — 대화형 이동 에이전트

  • 멀티턴 대화로 목적지·조건을 단계적으로 확정 (stateful)
  • 지도 위에서 경로·교통수단 후보를 비교 후 선택
  • 선택한 경로를 따라 단계별 안내
  • 환승 시점에 다음 이동수단까지 자연스럽게 연결
Section 03

서비스 흐름

서비스는 다섯 단계로 구성된다. 각 단계는 독립적으로 동작하면서도, 하나의 대화 세션 안에서 상태를 이어받아 누적된다.

1

멀티턴 대화로 목적지 확정

"병원 가야 하는데"처럼 불완전한 요청에서 시작해, 에이전트가 출발지·목적지·시간 제약을 단계적으로 채워간다. 한 번에 모든 정보를 요구하지 않는다.

난이도 낮음~중간
2

지도에 경로 표시

확정된 목적지를 기준으로 지도 위에 가능한 경로를 시각화한다. 버스·지하철·도보 등 복수 옵션을 한눈에 비교할 수 있다.

난이도 중간
3

선호 경로·교통수단 선택

사용자가 제시된 옵션 중 하나를 직접 선택한다. 에이전트가 추천하되 결정은 항상 사용자가 한다.

난이도 낮음
4

경로 안내

선택된 경로를 따라 다음 행동을 단계적으로 안내한다. 현재 위치 기준으로 "다음에 무엇을 해야 하는지"에 집중한다.

난이도 낮음~중간
5

환승 구간 연결

경로상 다음 교통수단으로 전환이 필요한 지점에 도착하면, 에이전트가 해당 수단 연결을 보조한다. 최종 실행 확정은 사용자가 직접 한다.

난이도 높음
Section 04

구현 범위 — 한 달 프로토타입 기준

본 프로젝트는 15주 일정의 학생 프로젝트로, 상용 서비스 출시가 아니라 컨셉 검증 가능한 프로토타입 구현을 목표로 한다. 외부 서비스와의 정식 제휴는 범위 밖이며, 필요 시 이후 단계에서 공단·KT를 통해 별도로 검토될 사항이다.

이번 단계에서 구현

  • 멀티턴 대화 기반 의도 파악 (슬롯 필링)
  • 공공 API 기반 버스·지하철 도착정보 조회
  • 2D 지도 위 경로 시각화 및 옵션 비교
  • 고정된 데모 환경(단일 기종·OS)에서의 환승 연결 시연

이번 단계에서 제외

  • 민간 서비스(카카오T 등)와의 정식 API 제휴
  • 다양한 기기·OS 환경에서의 범용 동작 보장
  • 결제·실행 단계의 완전 자동화

팀 회의 결정 필요 — 3D 지도 적용 여부
2D 경로 시각화는 구현 부담이 작고 일정 내 확실히 가능하다. 3D 지도(빌딩 렌더링, 카메라 틸트 등)는 시각적 임팩트가 크지만 별도 학습 곡선이 있어 1~2주가 추가로 소요될 수 있다. 발표가 "사내 공유회" 성격임을 고려하면 시각적 효과의 가치가 작지 않으므로, 풀 3D 구현 대신 핵심 구간만 3D로 보여주는 절충안도 함께 검토한다.

디자인 레퍼런스: Sinport — Singapore Airport navigation UI ↗ (카드형 내비게이션 레이아웃 참고용, 저작물 출처 표기)

역할 구분 — 에이전트는 탐색·비교·판단까지를 맡고, 결제나 최종 실행처럼 책임이 따르는 동작은 사용자가 직접 확정한다. 이 구분은 구현 난이도를 낮추는 동시에, 발표 과정에서 "무엇을 실제로 구현했는지"를 명확하게 설명할 수 있게 한다.

Section 05

단계별 난이도 분석

다섯 단계 중 1~4단계는 합쳐서 약 2주 내 프로토타입이 가능한 수준이다. 일정 리스크는 5단계(환승 구간 연결)에 집중되어 있다.

1. 멀티턴 대화로 목적지 확정

●●○○○

LLM function calling으로 출발지·목적지·시간 슬롯을 단계적으로 채우는 구조. 정립된 패턴으로 1주 내 프로토타입 가능.

2. 지도에 경로 표시

●●●○○

2D 경로 시각화는 지도 API로 1~2일 내 가능. 3D 렌더링은 학습 곡선이 있어 1~2주 추가 소요 — 적용 여부는 팀 회의에서 결정.

3. 선호 경로·교통수단 선택

●○○○○

공공 길찾기 API가 복수 경로 옵션을 반환하므로, 카드 UI로 노출하는 정도의 작업.

4. 경로 안내

●●○○○

실시간 턴바이턴 안내까지 가면 복잡해지나, "현재 위치 기준 다음 행동" 수준으로 범위를 좁혀 구현한다.

5. 환승 구간 연결

●●●●○

도착 감지(geofencing) 자체는 단순하나, 외부 앱 화면 제어가 필요해 별도 환경 구성이 필요하다. 고정된 데모 기종·시나리오로 범위를 한정해 1~2주 내 시연 가능한 수준으로 좁혔다.

5단계 구현 방식 검토

방식구현 방법한계채택 여부
딥링크 호출 특정 URL 스킴으로 외부 앱을 열어줌 앱을 여는 데서 끝남. 호출 결과를 돌려받을 수 없음 보조 수단
화면 제어 자동화 화면을 인식해 직접 탭·입력까지 수행 고정된 데모 기종·OS 버전에서만 안정적으로 동작 채택 (범위 한정)
정식 API 제휴 서비스사와 협의해 정식 연동 학생 프로젝트 범위를 넘어서는 별도 협의 필요 범위 밖
Section 06

차별점과 확장 가능성

  • 단발성 호출이 아닌 다단계 에이전트 — 입력을 음성에 한정하지 않고, 핵심 차별점을 상태 관리 기반의 연속적 의사결정 지원에 둔다.
  • 판단은 에이전트, 실행은 사용자 — 책임이 따르는 동작(결제·확정)은 항상 사용자가 직접 수행하도록 설계해, 신뢰 가능한 보조 구조를 갖춘다.
  • 공공데이터 기반 — 버스·지하철 도착정보는 국가교통정보센터·지자체 공공 API로 확보 가능해, 외부 제휴 없이도 핵심 기능 시연이 가능하다.
  • 후속 확장 경로 — 본 단계는 컨셉 프로토타입이며, 실제 서비스 연계나 민간 제휴는 공단·KT 차원의 후속 검토 사항으로 명확히 구분한다.

디자인 참고

아래 레퍼런스(Sinport — Singapore Airport Navigation)에서 가져올 핵심 디자인 요소를 정리한다.

[R1]

Sinport — Mobile App UI Design for Singapore Airport Navigation ↗

Dribbble · 디자이너 포트폴리오

가져올 디자인 요소

  • 큰 숫자형 단계 카드 — 게이트 번호, 남은 시간처럼 가장 중요한 정보 하나를 큰 숫자/큰 글자로 화면 상단에 고정 노출. 본 프로젝트에서는 "다음 정류장까지 N분", "환승까지 N분" 같은 값을 동일한 위치·크기로 고정해, 고령자가 화면을 스캔하지 않고도 한눈에 핵심만 확인하게 한다.
  • 경로를 따라가는 세로형 타임라인 — 출발지부터 목적지까지의 구간을 위에서 아래로 흐르는 카드 스택으로 표현하고, 현재 위치에 해당하는 카드만 강조색으로 부각. 4단계(경로 안내)·5단계(환승 연결) 화면에 적용해 "지금 어디까지 왔는지"를 텍스트 설명 없이도 인지하게 한다.
  • 한 화면 = 한 결정 — 레퍼런스는 화면마다 사용자가 내려야 할 결정을 하나로 제한하고, 나머지 정보는 보조적으로 흐리게 배치한다. 본 프로젝트의 3단계(경로·교통수단 선택)에서 후보를 2~3개 카드로만 제시하고, 선택 즉시 다음 화면으로 전환하는 구조에 그대로 적용한다.
  • 방향성 아이콘 + 최소 텍스트 — 화살표·도착지 핀 같은 단순 픽토그램을 본문 텍스트보다 크게 배치해, 문해력이나 시력에 관계없이 진행 방향을 인지하게 한다. 2단계(지도 경로 표시)에서 경로선과 함께 적용한다.
  • 고대비 색상 + 큰 터치 영역 — 레퍼런스의 버튼·카드는 터치 영역이 넓고 배경과의 명도 대비가 뚜렷하다. 고령자의 터치 정확도·시인성을 고려해, 본 프로젝트 전체 화면의 버튼 최소 높이와 텍스트-배경 대비비를 이 기준에 맞춰 통일한다.

적용 우선순위 — "큰 숫자형 단계 카드"와 "한 화면 = 한 결정" 두 요소가 고령자 사용성에 가장 직접적으로 기여한다. 프로토타입 제작 시간이 부족할 경우 이 두 가지를 먼저 구현하고, 세로형 타임라인과 아이콘 디테일은 이후 다듬는다.
