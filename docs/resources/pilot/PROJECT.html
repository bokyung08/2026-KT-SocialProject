Section 01

프로젝트 주제 설명

개인형이동장치(PM, Personal Mobility)로 인한 교통사고는 2020년 897건에서 2024년 2,232건으로 5년간 약 2.5배 증가했다. 사망자 수 역시 같은 기간 2.3배 늘어났다(한국교통안전공단, 2024). 특히 대전은 KAIST·충남대·한밭대·배재대 등 대학이 밀집한 도시로, 20대 PM 이용자가 집중되는 구조적 특성을 가진다.

기존 공유킥보드 앱(킥고잉·씽씽 등)은 주변 킥보드 위치 확인, QR 대여, 반납 구역 안내 기능만 제공하며 위험구간에 대한 실시간 정보는 전혀 제공하지 않는다. 사고는 언제나 일어나지만, 이용자는 어느 구간이 위험한지 알 방법이 없다.

핵심 문제 정의 — PM 사고 위험은 날씨·시간대·이동량 조건에 따라 동적으로 변하나, 현재 이용자에게 전달되는 안전 정보는 정적 규제(속도 제한, 반납 금지구역)에 국한되어 있다.

본 프로젝트는 날씨·시간대·이동량을 실시간으로 반영한 PM 사고 위험구간 예측 모델을 구축하고, 그 결과를 웹 대시보드(히트맵)와 모바일 위치기반 알림 서비스로 제공한다. 데이터는 한국교통안전공단 PM 사고 데이터, TAAS 교통사고분석시스템, 기상청 API, OpenStreetMap 도로 데이터를 결합하여 활용한다.

한 줄 요약 — 실시간 교통 상황(날씨, 시간대, 이동량)을 반영하는 PM 사고 위험구간 예측 대시보드(히트맵) 및 위치기반 진동 알림 서비스

본 프로젝트의 가장 직접적인 선행 사례다. 한국교통안전공단이 공유킥보드 서비스 빔모빌리티(Beam Mobility)와 협업해 PM 주행궤적 빅데이터 기반 위험구간 도출 시범사업을 수행했다(서울 5개구·청주·김천, 2023.10~2024.02). → 기사 원문 바로가기

시범사업의 주요 성과 — 전체 도로 구간 중 위험 핫스팟 2.7%(423곳) 도출. 위험주행 유형은 급가속 후 급감속 92.1%, 고속주행 후 급정지 7.3%. 대학가·학교·아파트 단지·상업지역에서 집중 발생.

Section 03

프로젝트 서비스 형태

서비스는 크게 두 가지 인터페이스로 제공된다. 웹 대시보드는 PM 이용자라면 누구나 접근할 수 있는 위험구간 히트맵이며, 앱에 진입하면 가장 먼저 보이는 메인 화면이다. 모바일 알림은 실제 주행 중 위험구간 진입 시 진동과 함께 경고를 받는 인터페이스다. 지자체·연구자·관리자는 동일한 대시보드를 정책 수립 용도로 활용할 수 있다.

① 위험구간 예측 모델

공단 PM 사고데이터 + 기상청 날씨 + OSM 도로 구조 + 시간대·요일을 결합한 머신러닝 모델. 구간별 위험도 등급(저·중·고) 산출. 모델 선택은 하단 비교 분석 참조.

② 웹 대시보드 (히트맵)

앱 메인 화면 및 웹에서 바로 접근. 대전 지도 위에 위험도를 색상으로 표시. 시간대·날씨 조건 슬라이더로 동적 시뮬레이션 가능. Folium / Kepler.gl 기반.

③ 모바일 위치기반 알림

GPS 위치가 고위험 구간 진입 시 진동+알림 발송. 백그라운드 위치 추적 및 푸시 알림 필요. 플랫폼 선택은 하단 분석 참조.

④ 서비스 확장 시나리오

대전시 교통국 정책 연계, 킥고잉·씽씽 API 파트너십, 충남대·KAIST 캠퍼스 안전팀 협력 등 지자체 상용화 경로 제시.

예측 모델 선택 분석 — XGBoost vs TabNet vs FT-Transformer vs 딥러닝

정형 교통 데이터(사고 이력, 날씨, 시간대, 도로 구조)를 피처로 사용하는 이 프로젝트에서 모델 선택은 예측 성능과 구현 공수의 균형이 핵심이다.

XGBoost 권장

정형 데이터 핫스팟 예측에서 검증된 성능. SHAP으로 피처 중요도 해석 가능 → 발표에서 "왜 이 구간이 위험한가" 설명 가능.

✓ 빠른 학습 · 해석 가능 · 소규모 데이터에 강함

✗ 시계열 패턴 직접 모델링 불가

성능 ★★★★공수 ★☆☆☆해석 ★★★★★
TabNet

어텐션 기반 정형 데이터 전용 딥러닝. XGBoost와 유사한 해석 가능성. PyTorch-TabNet 라이브러리로 구현 가능.

✓ 딥러닝 활용 명분 · 해석 가능

✗ 소규모 데이터에서 XGBoost 대비 성능 우위 불명확

LSTM은 시계열, GNN은 도로 그래프 구조를 직접 모델링. 이론적으로 가장 적합하나 구현 난이도가 가장 높음.

✓ 시공간 패턴 직접 모델링

✗ 15주 일정 내 완성도 확보 어려움 · 데이터량 필요

결론 — 1차 구현은 XGBoost + SHAP으로 빠르게 베이스라인 확보. 여유가 있을 경우 TabNet과 성능 비교 실험을 추가하면 "딥러닝 전공 역량 활용" 서술이 가능해진다. FT-Transformer·GNN은 데이터 규모와 일정을 확인한 후 판단.

모바일 플랫폼 선택 분석

백그라운드 위치 추적과 푸시 알림이 핵심 기능인 만큼, 플랫폼 선택이 구현 가능성에 직접 영향을 미친다.

옵션백그라운드 위치푸시 알림공수특이사항
KMP (Kotlin Multiplatform) 1순위 ✅ 네이티브 수준✅ FCM 연동높음 비즈니스 로직 공유, UI는 플랫폼별. JetBrains 공식 지원. 장기 유지보수성 우수.
Flutter 대안 검토 ✅ 네이티브 수준✅ FCM 연동중간 행정안전부 전자정부 프레임워크 연동 시 Flutter 기반 전자정부 SDK 존재 → 지자체 연계 서비스 확장 시 유리. Dart 학습 비용 있음.
React Native ✅ expo-location✅ expo-notifications중간 JS 생태계 친숙도 높으나, Expo Go 기반 프로토타입은 일부 백그라운드 제한 있음.
PWA 시간 부족 시 ⚠️ 제한적 (포그라운드만)⚠️ iOS 제한낮음 백그라운드 위치 추적 불가. 발표 데드라인 임박 시 최후 대안으로 채택. 핵심 기능 시연은 가능.

권장 전략 — KMP를 1순위로 검토. 정부 전자프레임워크(전자정부 표준프레임워크)와의 연동이나 지자체 상용화를 고려하면 Flutter가 더 유리할 수 있다. 팀 내 Kotlin 또는 Dart 숙련도를 확인 후 결정. 일정이 촉박할 경우에 한해 PWA 채택을 검토하되, 이 경우 "백그라운드 알림은 향후 네이티브 앱 전환 시 구현 예정"으로 발표 자료에 명시할 것.

기술 스택

  • 데이터 분석: Python (pandas, scikit-learn, XGBoost, geopandas, osmnx, pytorch-tabnet)
  • 웹 대시보드: Streamlit 또는 React + Folium / Kepler.gl
  • 모바일: KMP (1순위) / Flutter (정부 연계 고려 시) / PWA (일정 부족 시)
  • 지도 데이터: OpenStreetMap (osmnx, API 키 불필요)
  • 배포: Streamlit Cloud (대시보드), TestFlight / Firebase App Distribution (앱 프로토타입)

필요 API 및 데이터 소스

아래 API 및 데이터는 모두 검색 또는 킥오프 자료 교차검증을 통해 접근 가능성이 확인된 것만 기재한다. 공단 제공 데이터는 킥오프 자료 8페이지 목록과 교차검증 결과를 병기한다.

기상청 단기예보 조회서비스 API ↗

무료 즉시 신청 필요
  • data.go.kr 회원가입 후 "기상청_단기예보 조회서비스" 검색 → 활용신청
  • 승인까지 최소 1일 소요. 개발계정 일 10,000회 요청 제한.
  • 과거 기상 데이터(모델 학습용)는 기상자료개방포털(data.kma.go.kr)에서 별도 CSV 다운로드 필요
⚠ 실황이 아닌 3시간 단위 예보값 제공. 2년마다 API 키 갱신 필요. 위경도 → 기상청 격자(nx/ny) 변환 엑셀 파일 별도 다운로드 필요.

TAAS 교통사고 데이터 ↗

무료
  • 건별 원시 사고 데이터: taas.koroad.or.kr → 통계정보 → 수단별 → 개인형이동장치 → 대전 필터 → 파일 다운로드 (REST API 아님)
  • 사고다발지 위치 정보 API: opendata.koroad.or.kr 회원가입 후 신청
  • 대전 PM 특화 데이터: 공공데이터포털 data.go.kr/data/15123495 — 경찰청 대전광역시경찰청 PM 법규위반별 교통사고(2018~2023), 로그인 없이 즉시 다운로드 가능

OpenStreetMap 도로 데이터 (osmnx)

무료 API 키 불필요
  • pip install osmnx 후 ox.graph_from_place("Daejeon, South Korea") 한 줄로 대전 도로망 추출
  • Overpass API 호출 방식 — 대용량 추출 시 서버 부하 주의, 시간대 분산 요청 권장

공단 제공 데이터 (이메일 요청)

별도 신청 즉시 발송 필요
  • 담당자: 조용빈 선임연구원 — selisia@kotsa.or.kr
  • 데이터 요청서 + 보안서약서 제출 필요. 저용량은 메일, 대용량은 실물 저장장치 수령.
✅ 킥오프 자료(8p) 교차검증 완료 — 요청 가능 항목
① 첨단모빌리티 현황분석데이터 — "자율주행, DRT, PM, 플랫폼택시 등 첨단모빌리티의 이용, 운행 등(비정형데이터)" → PM 사고·운행 데이터 포함. 요청서에 PM 한정 명시 권장.
② 단지내도로 교통안전 실태점검데이터 — "도로외구역인 대학교·아파트 내 도로에 대한 안전실태점검(비정형데이터)" → 킥오프 자료에 명시 확인.
③ 교통문화지수데이터 — "지자체별로 조사된 운전행태, 교통안전, 보행행태 등(데이터 및 비정형데이터)" → 킥오프 자료에 명시 확인.
④ 교통카드 빅데이터 — "대중교통 이용량, 통행시간, O/D지표 등(데이터)" → 이동량 피처 활용 가능. 대용량으로 실물 저장장치 수령 예상.
⚠ 주의 — 킥오프 자료에 "PM 사고데이터"라는 독립 항목은 없음. PM 사고 데이터는 "첨단모빌리티 현황분석데이터" 또는 "교통안전특별실태조사데이터" 내에 포함될 가능성이 높음. 요청서 작성 시 "PM 관련 사고 및 운행 데이터"로 명확히 기재하고, 어느 데이터셋에 포함되어 있는지 공단 담당자에게 사전 확인 권장.
