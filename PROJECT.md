# 개인형 이동장치(PM) 연속주행 안전 경로 탐색 서비스

## 1. 프로젝트 개요

### 1.1 서비스 개요
개인형 이동장치(전동킥보드, 자전거 등) 이용자를 위한 연속주행 안전 경로 탐색 서비스.

### 1.2 추진 배경
전동킥보드와 자전거 등 개인형 이동장치 이용자는 이동 과정에서 자전거도로 단절, 차도 합류, 교차로 진입, 보행자와의 혼재 등 다양한 위험 상황을 경험한다.

현재 대부분의 지도 서비스는 최단거리·최단시간 중심으로 경로를 제공하여, PM 이용자에게 적합하지 않은 도로가 경로에 포함되거나 자전거도로가 중간에 끊겨 차도로 진입해야 하는 문제가 발생한다.

기존 PM 안전 대책(헬멧 착용, 속도 제한, 주차 단속 등)은 이용자 행동 관리에 집중되어 있으나, 위험한 주행 행동은 이용자 부주의뿐 아니라 자전거도로 단절 같은 **도로 구조적 요인**으로도 발생한다.

본 프로젝트는 이용자 행동 규제 대신, 위험한 행동을 유발할 수 있는 도로 구조를 분석하여 최대한 연속적이고 안전하게 주행할 수 있는 경로를 제공하는 것을 목표로 한다.

### 1.3 문제 정의
기존 경로 탐색 서비스의 한계:
1. 최단거리/최단시간 우선으로 자전거도로 연결 상태가 충분히 반영되지 않음
2. 자전거도로 단절을 주요 위험 요소로 평가하지 않음
3. 차도 합류, 교차로, 사고 다발 구간 등 위험 도로 구조가 경로 탐색 비용에 충분히 반영되지 않음
4. 경로별 위험 요소와 자전거도로 연속성을 정량적으로 비교할 기준 부족

### 1.4 프로젝트 목표
출발지·목적지를 입력받아 PM 이용자가 최대한 연속적이고 안전하게 주행할 수 있는 경로를 추천.

세부 목표:
- 자전거도로 연결 상태를 반영한 도로 분석
- 자전거도로 단절·차도 합류 구간 탐지
- 도로 유형·위험 요소별 비용값 설계
- A*(변경 가능) 기반 안전 경로 탐색 알고리즘 구현
- 복수 후보 경로 생성 및 경로별 안전성 비교
- 경로별 연속주행 안전 점수 산출
- 웹 지도에서 추천 경로·위험 구간 시각화

### 1.5 핵심 서비스 개념
**연속주행**: 출발지부터 목적지까지 자전거도로가 최대한 유지되고, 차도·보도·자전거도로 간 불필요한 전환이 적은 상태.

- 거리가 짧아도 자전거도로가 여러 차례 단절되고 차도 반복 진입이 필요한 경로 → 연속성 낮음
- 거리가 다소 길어도 자전거도로가 지속 연결되고 차도 합류·위험 교차로 통과가 적은 경로 → 연속성·안전성 높음

**비용함수 고려 요소 (가중치형 변수)**
- 이동 거리
- 자전거도로 이용 비율
- 자전거/보행자 겸용 도로 시 차도 사용 자제 여부
- 도로 유형 전환 횟수 (자전거도로 단절 / 차도 합류 횟수)
- 교차로·횡단보도 통과 횟수
- 버스/트램 노선과의 중첩 여부
- (기각됨, 아래 §3 참고) 사고 위험 구간 포함 여부

**OX형 변수**
- PM 통행 가능 여부 (교차로 1차선 강제 이용 등)

## 2. 경로 탐색 알고리즘 설계

### 2.1 NP-hardness 회피 원칙
다중 변수(거리, 자전거도로비율, 전환횟수, 교차로통과 등)를 **개별 하드 제약(hard constraint)**으로 다루면 Multi-Constrained Shortest Path(MCSP) 문제가 되어 NP-complete가 된다. Pareto-optimal 경로 집합 전체를 구하는 접근도 최악의 경우 해 개수가 지수적으로 증가해 다루기 어렵다.

**해결 원칙 — P로 유지하는 설계:**

| 문제 상황 | 하지 말아야 할 것 | 해야 할 것 |
|---|---|---|
| 다중 변수 반영 | 여러 변수를 개별 하드 제약(≤N)으로 걸기 | 가중합(weighted sum)으로 스칼라 비용 하나로 합치기 |
| 복수 후보 경로 필요 | 전체 Pareto 최적해 집합 구하기 (해 개수 지수적 폭발 위험) | k-최단경로(Yen's Algorithm 등)로 top-k만 정확히 추출 — 근사가 아닌 엄밀한 순위 계산 |
| 직전 상태 의존 비용 (전환 페널티 등) | "직전 상태 의존" 비용을 노드 단위로만 계산 (다익스트라 전제 붕괴) | (노드, 직전 도로유형) 상태로 그래프 확장 |
| PM 통행 불가 구간 (OX형 변수) | 통행 불가를 제약조건으로 별도 처리 | 통행 불가 엣지는 그래프에서 아예 제거 (전처리) |

> 참고: k-최단경로(Yen's)는 근사 알고리즘이 아니라, 주어진 스칼라 비용함수 기준으로 정확히 1등~k등을 찾는 엄밀한(exact) 알고리즘이다. k=1이면 일반 A*/Dijkstra와 동일한 최적해와 일치한다. 여기서 실제로 포기하는 것은 "여러 원시 기준(거리, 전환횟수 등)을 동시에 최적화하는 Pareto 해 집합 전체"이며, 이는 애초에 위 첫 번째 행(가중합으로 스칼라화)에서 문제 자체를 우회했기 때문에 발생하지 않는다.

### 2.2 비용함수 형태
```
cost(edge) = w1*거리 + w2*(차도구간 페널티) + w3*(전환 페널티)
           + w4*(교차로/횡단보도 페널티) + w5*(버스노선 겹침 페널티)
```
엣지마다 음이 아닌 실수 가중치 하나를 갖는 표준 그래프로 스칼라화 → 다익스트라/A*로 다항 시간 내 최적해 보장.

### 2.3 상태 확장 (직전 도로유형 반영)
"도로 유형 전환 횟수"처럼 직전에 어떤 도로를 지나왔는지에 따라 현재 엣지 비용이 달라지는 경우, 순수 다익스트라의 전제(노드별 최단거리는 독립적)가 깨진다.

**해결책**: 상태를 `(현재 노드)` 대신 `(현재 노드, 직전 도로유형)`으로 확장.
- 그래프 크기가 `V → V × k` (k = 도로유형 개수, 보통 3~5개)로 증가하지만 여전히 다항 크기 → A*/다익스트라 그대로 적용 가능
- line graph(엣지 기반 그래프) 변환과 동일한 원리

**A* 적용 가능성**: A*의 요구사항(그래프 구조 + admissible 휴리스틱)은 상태 확장 후에도 그대로 성립하므로 문제없이 사용 가능.

**휴리스틱 설계 원칙**: g(누적비용)는 전환 페널티·위험도를 다 반영하되, h(휴리스틱)는 **순수 거리 기반 하한값만** 사용해야 admissibility가 보장됨. 전환 페널티처럼 경로에 따라 발생 여부가 달라지는 값을 h에 넣으면 과대추정 위험 → 최적해 보장이 깨질 수 있음.

**상태 폭발 관리**: k(도로유형 개수)가 3~5개 수준이면 문제없음. "직전 N개 세그먼트 조합"까지 반영하면 `V×k^n`으로 커지므로, 본 프로젝트 스코프에서는 **직전 1개 세그먼트**로 제한 권장.

### 2.4 복수 후보 경로 생성
Pareto 멀티오브젝티브 대신 **Yen's Algorithm** 등 k-최단경로 알고리즘 사용 (다항 시간, O(kV(E+V log V)) 수준). 단일 스칼라 비용함수 기준 top-k 후보를 뽑고, 후보들에 대해서만 안전 점수를 별도로 비교.

## 3. 경로 탐색 엔진 — GraphHopper 활용

### 3.1 채택 근거
GraphHopper는 오픈소스 라우팅 엔진(Apache 2.0)으로, Dijkstra/A* 및 양방향 변형을 기본 지원하며, 엣지별 비용함수를 코드 또는 JSON 설정(`custom_model`)으로 커스터마이징 가능. §2에서 설계한 인프라(A*, 상태확장, 커스텀 비용함수)를 직접 구현하는 대신 라이브러리로 재사용.

### 3.2 구현 범위 (직접 구현 vs 기본 제공)
| 필요한 작업 | 직접 구현 여부 |
|---|---|
| A*/Dijkstra 탐색 알고리즘 | ❌ 기본 제공 |
| 그래프 상태를 (node, prev_type)으로 확장 | ❌ turn cost(edge-based routing) 메커니즘이 대체 처리 |
| 엣지별 비용함수 | ✅ `Weighting` 클래스 구현 (`calcEdgeWeight`, `getMinWeight`) |
| 전환 페널티 (도로유형 전환 시) | ✅ `TurnCostProvider` 클래스 구현 (`calcTurnWeight(inEdge, viaNode, outEdge)`) |
| 위 클래스 등록 | ✅ `WeightingFactory` 오버라이드 |

turn cost가 켜지면 GraphHopper는 내부적으로 edge-based 탐색을 수행하여, 사실상 `(node, prev_edge)` 상태 확장과 동일한 효과를 이미 인프라 레벨에서 제공한다. 즉 §2.3의 상태 확장을 별도 그래프 뻥튀기 코드로 구현할 필요가 없다.

### 3.3 구현 언어
Kotlin으로 구현 가능. JVM 언어이므로 GraphHopper(Java 라이브러리)와 100% 상호운용.
- `AbstractWeighting` 상속, `TurnCostProvider` 구현은 Kotlin `override fun`으로 자연스럽게 작성
- 주의사항: GraphHopper Java API의 플랫폼 타입(`@Nullable` 어노테이션 불일치)으로 인한 NPE 가능성 → 명시적 null 처리 필요
- IRL로 학습한 가중치(w1~w5)는 data class로 관리 후 생성자 주입

### 3.4 속도 모드(CH/LM) 관련 트레이드오프
- Contraction Hierarchies(CH): 쿼리 속도 극대화하지만 사전 정의된 프로필만 가능, 준비 시간·자원 소모 큼
- Landmarks(LM, hybrid mode): 런타임 커스텀 모델 변경에 유연하지만, 가중치는 "증가만 가능, 감소 불가"라는 제약 존재
- **권장**: 프로토타입/가중치 튜닝 단계 = flexible 모드(자유도 최대), 가중치 안정화 후 배포 시점 = CH/LM 전환

## 4. 가중치 학습 — Inverse Reinforcement Learning(IRL) 프레이밍

### 4.1 정의
IRL: 전문가의 행동/선택 데이터(observed preference)로부터 그 행동을 만들어낸 보상(비용)함수 자체를 역산하는 문제.

| IRL 용어 | 본 프로젝트 대응 |
|---|---|
| 전문가의 행동(expert demonstration) | 경로 A/B 중 선호되는 쪽에 대한 라벨 |
| 보상함수(reward function) | 비용함수 가중치 w1~w5 |
| 학습된 보상함수로 정책을 다시 품 | 학습된 w로 A*/Dijkstra 실행 |

### 4.2 학습 방법 — Bradley-Terry 랭킹 로스
```
P(A ≻ B) = σ(w·x_A - w·x_B)
Loss = -Σ log P(선택된 경로 ≻ 비선택 경로)
```
로지스틱 회귀와 동일한 형태의 convex optimization → 안정적 학습. `w ≥ 0` 제약을 걸면 해석 가능한 가중치 산출. Max-Margin Planning / Structured SVM 기반 IRL과 동일 계열이며, 자율주행 경로비용 학습에서 흔히 쓰이는 방식.

**중요**: 가중치 학습(오프라인) 이후 실제 서비스에서 A* 실행 시에는 어떤 형태의 FM/모델 호출도 없음. cost function은 고정된 선형함수 `w1*x1 + ... `이라 탐색 속도에 영향 없음. 학습(느려도 됨)과 추론(빨라야 함)이 완전히 분리됨.

### 4.3 검토했으나 기각/보류한 방법들 (기록용)

#### (a) LLM(Claude/GPT)에게 직접 경로쌍 선호를 묻는 방식 — 초기안, 근본적 한계 존재
- 방식: FM에게 "경로 A: 거리 3.2km, 자전거도로비율 80%... vs 경로 B: ..." 형태로 feature를 제시하고 선호 판단 요청 → 선호 데이터셋 생성 → Bradley-Terry 학습
- **한계**: LLM은 실제 사고 통계나 한국 도로 구조를 학습한 적이 없어 "grounding 없는 판단"에 해당. 방향성(sign, 예: 교차로 통과가 위험하다)은 맞을 수 있으나 정량적 크기(magnitude, 예: 교차로 1회 = 거리 몇 m와 등가)를 신뢰할 근거가 없음. 동일 질문도 프롬프트 표현에 따라 답이 바뀔 수 있는 일관성 문제도 존재.
- **결론**: 실사고 데이터가 없는 콜드스타트 상황의 임시 prior 생성용으로만 제한적으로 고려 가능하나, 최종 채택하지 않음.

#### (b) 지리공간 파운데이션 모델 (Prithvi, Clay) — 기각
- 검토 배경: 사고 위험 구간을 이미지 기반으로 직접 탐지하려는 목적 및 IRL의 "선호 판단자" 역할 대체 가능성 검토
- **Prithvi** (NASA/IBM): Landsat/Sentinel-2 기반 HLS 데이터로 사전학습, 해상도 10~30m급. 최신 버전 Prithvi-EO-2.0(2026)은 6억 파라미터.
- **Clay**: Sentinel-1/2, Landsat-8, MODIS, NAIP 등 다중 센서 기반 ViT, 유사한 해상도 특성.
- **기각 사유 1 (해상도 미스매치)**: 자전거도로 단절, 차도 합류 등은 1~2m 단위 디테일인데, 10~30m 해상도로는 도로 구조 자체가 픽셀 몇 개로만 표현되어 판별 불가. Transformer 구조 특성상 압축 과정에서 해상도가 추가로 4~5배 손실되어 픽셀 단위 분할에 원래도 약함.
- **기각 사유 2 (역할 부적합)**: 두 모델 다 이미지를 임베딩 벡터로 변환하는 인코더로, "경로 A와 B 중 뭐가 안전한가"를 텍스트로 비교/판단하는 능력 자체가 없음 → IRL의 "선호 판단자(Step 2)" 역할 대체 불가.
- **기각 사유 3 (self-supervised의 오해)**: MAE(Masked Autoencoder) 방식의 self-supervised 학습은 "라벨 없이 위험도를 알아서 판단"해주는 것이 아니라 일반적 지표 패턴(토지피복 등)을 학습한 것 → 위험도 축을 담고 있다는 보장 없음, 결국 별도 라벨이 필요해 "사고 데이터 없음" 문제를 해결해주지 않음.
- **부분적으로 고려 가능했던 대안**: Clay가 지원하는 NAIP(0.6~1m 해상도, 미국 전용)라면 정밀도는 개선되나 한국 데이터 커버리지 없음 → 국내 고해상도 항공/드론 정사영상 별도 확보·파인튜닝 필요, 사실상 원 모델의 제로샷 장점 상실.

#### (c) 규칙 기반 대안: Level of Traffic Stress (LTS) — 참고용, 미채택이나 유효한 대안으로 기록
- 사고 데이터 없이도 도로 구조적 속성(차선 수, 자전거도로 폭, 제한속도, 주차차로 유무, 교차로 유형 등)만으로 규칙 기반 점수를 산출하는 교통공학 표준 방법론.
- OSM/표준노드링크 속성 데이터만으로 계산 가능, FM 불필요.
- 학부/대학원 발표 시 "왜 이 방법을 썼는가"에 대한 문헌 기반 근거가 탄탄함. 최종 채택된 (d) 방법과 병행 검토 가능한 대안으로 기록.

### 4.4 최종 채택 방법 — Street-view 기반 파인튜닝 + 지식 증류

**핵심 통찰**: ZenSVI 출력을 비용함수의 **추가 feature로 넣는 것이 아님**. TAAS 실사고 데이터로 파인튜닝한 이미지 기반 위험도 예측 모델을 **teacher**로, OSM feature 가중합 비용함수를 **student**로 두는 지식 증류 + IRL 하이브리드 구조. 가중치 학습 시에만 로드뷰 데이터가 필요하고, 학습 완료 후 배포 시에는 OSM 데이터만으로 전국 어디서든 동작 가능.

**ZenSVI**: 거리뷰 이미지 기반 도시 지각(perception) 분석 오픈소스 파이썬 패키지. Vision Transformer 기반 내장 모델이 안전 인지 등 6개 지각 차원에 대해 0~10점 점수 산출 (평균 정확도 61.6~77.1%, 차원별 상이). Global Streetscapes(Hou et al., 2024) 모델도 벤치마킹용으로 포함.

이 방법론은 MIT Place Pulse 2.0(쌍대비교 기반 안전 인지 데이터셋)의 학습 구조와 동일 계열이며, 본 프로젝트의 Bradley-Terry 기반 IRL 설계와 방법론적으로 일치.

**전체 파이프라인:**
```
1. 데이터 수집
   - TAAS(교통사고분석시스템) 대전 지역 PM/자전거 사고 지점 GIS 데이터 수집
     (koroad.or.kr, 지점 단위 조회/다운로드 가능, 출처 명시 필요)
   - 사고지점 + negative 지점(§4.5 참고)에 대해 네이버 로드뷰 등 이미지 수집

2. Teacher 모델 파인튜닝
   - ZenSVI 백본(ViT) 활용, 데이터 희소성 고려해 backbone 고정 + 소형 헤드만 학습 권장
   - 목표: 로드뷰 이미지 → 사고 확률(위험도) 예측

3. 전역 스코어링 (대전 전체, 1회성)
   - 대전 전체 OSM edge에 대해 로드뷰 수집
   - 파인튜닝된 teacher 모델로 edge별 risk_score 산출

4. Route risk 집계 (§5 참고)
   - edge별 risk_score를 route 단위로 집계하는 함수 필요 (설계 진행 중, 확정 안 됨)

5. IRL 가중치 학습
   - 임의의 경로쌍(A, B) 샘플링
   - route_risk(A) vs route_risk(B) 비교 → 선호 라벨 생성
   - Bradley-Terry 로스로 OSM feature 가중합 비용함수의 w1~w5 학습
   - 이 단계 이후 teacher 모델(이미지/로드뷰) 완전히 손을 뗌

6. 배포
   - 학습된 w를 GraphHopper custom Weighting에 주입
   - 실제 서비스는 OSM 데이터만으로 동작, 로드뷰 수집 불필요
```

### 4.5 남은 설계 이슈 (미확정, 다음 논의 필요)

**(1) 클래스 불균형**
대전 단일 도시, PM/자전거 특정 사고로 한정 시 표본 수가 적을 가능성 높음(수십~수백 건 수준 추정). 대응: backbone frozen + linear probe 방식의 few-shot에 가까운 학습, 필요 시 인접 도시(세종·청주 등) 표본 확장 검토.

**(2) Negative 샘플링 전략 — 미확정**
단순 랜덤 OSM 지점에서 negative를 뽑으면 "노출(exposure) 편향" 발생 위험 (통행량 많은 곳이 사고도 많고 이미지도 번화가처럼 보여 모델이 실제 도로 위험구조 대신 "번화가처럼 보이는지"를 학습할 위험). 통행량 매칭 negative 샘플링 등 구체 전략 확정 필요.

**(3) Route risk 집계함수 — 미확정, 후보 비교만 완료**
검토된 후보 (모두 기억해둘 것, 최종 선택은 보류):

| 방법 | 수식 | 장단점 |
|---|---|---|
| 평균 | $\frac{1}{n}\sum p_i$ | 위험구간이 경로 길이에 희석됨 (연속주행 철학과 불일치) |
| 최댓값 | $\max(p_i)$ | 위험구간 1곳 vs 여러 곳 구분 못함 (정보 손실) |
| 합 | $\sum p_i$ | 경로 길이(엣지 개수)에 단순 비례하는 왜곡 |
| 임계값 초과 개수 | $\sum \mathbb{1}[p_i > \tau]$ | 임계값 $\tau$ 설정이 자의적, 확률 크기 정보 손실 |
| **Hazard rate 기반 생존모델 (현재 추천안)** | $route\_risk = 1-\exp(-\sum h_i L_i)$, $h_i = -\ln(1-p_i)/L_i$ | 위험도 누적 반영 + 엣지 길이 불균일 보정 + 확률론적 해석 명확. 단, 확률 보정(calibration) 필요하고 경로 단위 ground truth 부재로 완전한 검증은 불가 |

부가 설계 요소:
- 파인튜닝 모델 출력의 overconfidence 보정 필요 (Platt scaling/isotonic regression)
- OSM edge 경계와 로드뷰 수집 단위 불일치 문제 → 고정 간격(30~50m) 샘플링 후 edge 매핑 방식 권장
- 검증 데이터 부재 문제 → 소규모(20~30쌍) 사람 투표로 후보 집계함수 간 정합성 sanity check 제안됨 (미실행)

**(4) 로드뷰 촬영 시점과 사고 발생 시점의 시간차**
도로 구조 변경(자전거도로 신설 등) 가능성. 완화책으로 지나치게 오래된 사고 데이터 필터링 고려.

**(5) 네이버 로드뷰 API 이용약관**
대량 이미지 수집이 이용약관상 허용되는 범위인지 확인 필요 (미확인).

## 6. 기각된 접근법 요약 (재검토 방지용 기록)

- ❌ 다중 변수를 하드 제약으로 건 경로탐색 → NP-hard화
- ❌ 전체 Pareto-optimal 경로 집합 계산 → 지수적 해 폭발
- ❌ 사고위험구간을 실시간 FM 호출로 판단 → 속도/비용 문제, 확정적 판단 불가능한 영역
- ❌ LLM에게 경로 feature 숫자를 주고 직접 선호 판단 요청 → grounding 부족, 일관성 문제
- ❌ Prithvi/Clay 등 위성 기반 지리공간 FM → 해상도 미스매치(10~30m ≫ 필요한 1~2m), 선호판단 능력 자체 없음
- ❌ ZenSVI 출력을 비용함수의 추가 feature로 직접 편입 → 배포 시 전국 로드뷰 수집이 필요해져 확장성 상실 (올바른 설계는 IRL 가중치 학습에만 한정 사용)
